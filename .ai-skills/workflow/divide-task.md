# divide-task

## 목적

현재 작업을 단일 에이전트가 계속 처리할지, 여러 작업 흐름 또는 하위 에이전트로 나눌지 판단합니다.
분배가 유리하고 실행 권한이 있으면 메인 에이전트가 직접 하위 에이전트를 생성해 작업을 맡기고, 결과를 통합합니다.

## 사용 시점

- 사용자가 큰 작업을 나눌지 판단해 달라고 할 때
- 멀티 에이전트, 병렬 작업, 업무 분배, 작업 패킷을 요청할 때
- 긴 논의 내용을 실행 가능한 하위 작업이나 하위 에이전트 작업 지시로 정리해야 할 때
- 작업 범위가 넓고 소유 범위, 수정 범위, 검증 경계가 나뉠 수 있을 때

## 핵심 원칙

### 단일 에이전트 우선

- 작업이 작거나 강하게 얽혀 있으면 단일 에이전트를 우선합니다.
- 다음 한 단계가 전체 핵심 경로라면 메인 에이전트가 직접 유지합니다.
- 병렬화로 줄어드는 시간보다 작업 지시 작성, 결과 통합, 충돌 조율 비용이 크면 나누지 않습니다.

### 실행 권한

- 사용자가 `$divide-task`를 명시적으로 호출했고 실제 작업 수행을 요청했다면 병렬 하위 에이전트 실행을 허가한 것으로 봅니다.
- 사용자가 "계획만", "프롬프트만", "나누기만", "실행하지 말고"처럼 실행 제외를 말하면 실제 하위 에이전트를 실행하지 않습니다.
- 실행 가능한 환경에서는 사용자에게 복붙용 스크립트나 프롬프트 실행을 맡기지 않고, 메인 에이전트가 직접 하위 에이전트를 생성합니다.
- 하위 에이전트 실행 권한이나 도구가 없을 때만 붙여넣기용 프롬프트 또는 실행 스크립트를 대안으로 제공합니다.

### 소유 범위

- 코드 변경은 수정 범위가 겹치지 않게 나눕니다.

## 절차

### 판단

1. 목표와 현재 상태를 2-4줄로 재정의합니다.
2. 다른 작업보다 먼저 끝나야 하는 핵심 경로를 찾습니다.
3. 작업을 차단 작업, 병렬 보조 작업, 독립 구현 단위, 검증/리뷰로 나눕니다.
4. 멀티 에이전트로 나눌 가치가 있는지 판단합니다.
5. `single-agent`, `single-agent-with-side-notes`, `multi-agent-plan-only`, `multi-agent-execute` 중 하나를 선택합니다.

### 분배와 통합

1. 나눌 경우 각 작업 흐름의 소유 범위와 수정 금지 범위를 정합니다.
2. `multi-agent-execute`이면 메인 에이전트가 직접 하위 에이전트를 생성하고 작업 지시를 전달합니다.
3. `multi-agent-plan-only`이거나 실행이 불가능한 환경이면 하위 에이전트 지시문과 통합/검증 절차만 작성합니다.
4. 하위 에이전트 결과를 검토해 충돌, 누락, 검증 실패를 메인 에이전트가 통합 처리합니다.

### 판단 기준

아래 조건 중 2개 이상이면 멀티 에이전트 분배를 고려합니다.

#### 병렬성

- 병렬 진행 가능한 독립 영역이 3개 이상입니다.
- 논의/요구사항이 길어서 작업 패킷으로 요약해야 실수가 줄어듭니다.
- 기획, 디자인, 프론트엔드, 백엔드, 모바일, 데이터, QA, 릴리즈처럼 산출물 소유 범위가 명확히 갈립니다.

#### 소유와 검증

- 구현이 진행되는 동안 병렬로 돌릴 수 있는 읽기 전용 조사 작업이 있습니다.
- 파일/모듈 수정 범위가 서로 겹치지 않습니다.
- 구현이 끝나기 전에도 병렬 검증/리뷰가 가능합니다.

아래 조건 중 하나라도 강하면 나누지 않습니다.

- 다음 행동이 하나의 미확정 답변에 막혀 있습니다.
- 두 에이전트가 같은 파일이나 같은 계약을 수정하게 됩니다.
- 작업이 대부분 판단, 제품 방향성, 아키텍처 선택입니다.
- 사용자가 실행하지 말고 답변이나 계획만 요청했습니다.
- 작업 지시 작성/결과 통합 비용이 실제 작업보다 큽니다.

### 일반 분배 패턴

프로젝트 종류에 따라 아래처럼 작업 흐름을 나눌 수 있습니다. 실제 분배는 현재 요청의 산출물, 파일 소유권, 검증 방식에 맞춰 조정합니다.

```text
제품/기획
-> 문제 정의, 요구사항, 범위, 성공 기준, 리스크 검토

디자인
-> UX, UI 구조, 디자인 계약, 상태/상호작용 정의

프론트엔드
-> 화면 구현, 상태 관리, API 연동, 접근성, UI 테스트

백엔드
-> API, 데이터 모델, 인증/권한, 비즈니스 로직, 통합 테스트

모바일
-> Android/iOS/React Native/Flutter 구현, 플랫폼 기능, 스토어 제약

데이터/분석
-> 이벤트 설계, 지표, 데이터 파이프라인, 대시보드

QA/검증
-> 테스트 계획, 회귀 검증, 인수 기준, 차단 이슈 관리

DevOps/릴리스
-> CI/CD, 배포, 릴리즈 체크리스트, 운영 모니터링

문서/기술 작성
-> 사용자/개발자 문서, 운영 절차서, 인수인계 문서, 변경 기록
```

## 출력

### 답변 형식

작업이 아주 작지 않다면 아래 형식으로 답합니다.

```text
결정
단일 에이전트 / 멀티 에이전트 판단과 한 문장 근거.

이유
핵심 경로, 결합도, 리스크, 조율 비용.

작업 분배
에이전트/작업 흐름 이름, 목적, 소유 범위, 입력, 출력.

실행 방식
메인 에이전트가 직접 실행할 하위 에이전트, 또는 실행하지 않고 계획만 제공하는 이유.

하위 에이전트 지시문
메인 에이전트가 하위 에이전트에 전달할 작업 지시. 사용자에게 복붙을 요구하지 않습니다.

통합 계획
메인 에이전트가 결과물을 수집, 리뷰, 병합, 검증하는 방법.

검증
완료 전 확인할 명령, 테스트, 리뷰 기준, 문서.
```

### 하위 에이전트 지시문

하위 에이전트 지시문은 아래 정보를 포함합니다.

```text
목표:
맥락:
소유 범위:
수정 금지:
입력:
필수 출력:
검증:
최종 응답:
```

### 위임 문서 위치

실제 하위 에이전트를 실행하지 않고 작업 패킷 문서가 필요할 때만 repo 성격에 맞는 임시 문서 위치를 제안합니다.

우선순위:

```text
1. 이미 프로젝트에 작업/run/plan 폴더가 있으면 그 구조를 따른다.
2. 없으면 docs/delegation/{YYYYMMDD}-{task-slug}.md 를 제안한다.
3. 파일을 만들지 말라는 요청이면 답변 안에만 작업 패킷을 작성한다.
```

최종 답변에는 아래를 포함합니다.

- 문서 경로
- 어떤 에이전트가 어느 작업을 받았는지
- 하위 에이전트 실행 여부와 실행하지 않았다면 그 이유
- 기대 산출물
- 통합 순서

## 점검

### 분배 판단

- 왜 나누는지 또는 왜 나누지 않는지 설명했는가
- 두 에이전트가 같은 수정 범위를 소유하지 않는가
- 명시적으로 위임하지 않은 핵심 경로는 메인 에이전트가 유지하는가

### 실행과 통합

- 실제 실행 요청이 있으면 복붙 안내에 그치지 않고 가능한 하위 에이전트를 직접 실행했는가
- 모든 하위 에이전트 지시문이 독립 실행 가능한 맥락을 갖는가
- 통합/검증 계획이 있는가
- 사용자가 실행 제외를 요청했는데 하위 에이전트를 실행하지 않았는가
